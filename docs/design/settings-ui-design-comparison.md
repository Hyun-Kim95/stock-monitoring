## 디자인 관점 장단점 요약

### 안 A (로컬 와이어 + 정적 목업)

| 장점 | 단점 |
|------|------|
| **제품과 동일 토큰**(`globals.css` OKLCH 축약)으로 라이트/다크·대비를 **구현 기준에 바로 맞출 수 있음** | **픽셀 단 polish**는 개발자·디자이너 손이 더 듦 |
| **IA·상태(로딩·빈·오류)**를 표·와이어로 **명시**해 QA·PRD와 정합시키기 쉬움 | 이해관계자에게 줄 때 “**고급 시안**” 인상은 약할 수 있음 |
| **오프라인·키 없이** 즉시 공유 가능 | 테마·뉴스 규칙·설정 화면은 **종목 수준의 디테일**보다 요약형 |
| PRD **Q4(테이블 페이지네이션)** 갭을 문서에 **그대로 드러내** 조정 논의 가능 | Stitch 대비 **시각적 대안 탐색**(여러 무드)이 느림 |

### 안 B (Stitch 생성)

| 장점 | 단점 |
|------|------|
| **한 번에 시각 시안**(레이아웃·밀도·금융 SaaS 톤) 확보 → 리뷰·공유에 유리 | **실제 앱 토큰/컴포넌트**와 1:1 일치는 보장 안 됨 → 이행 시 **조정 작업** 필요 |
| 프롬프트만 바꿔 **변형·재생성**으로 방향 탐색이 빠름 | **API 키·네트워크** 의존, 생성 시간·한도 이슈 가능 |
| 스크린샷·HTML URL로 **스테이크홀더용 자료** 만들기 쉬움 | **다크·모바일**은 별도 생성·디자인 시스템 단계가 필요할 수 있음 |
| “깔끔한 관리 콘솔” **첫인상** 정렬에 강함 | PRD의 **세부 상태(409 문구 등)** 는 코드/안 A와 **수동 매핑** |

## 비교표 (요약)

| 항목 | 안 A (로컬) | 안 B (Stitch CLI) |
|------|-------------|-------------------|
| **시각 임팩트** | 기능·구조 중심 | 시안·무드 중심 |
| **제품 일치도** | 높음(토큰·패턴 동일 계열) | 중간(이행 시 튜닝) |
| **상태 UI 스펙** | 표·목업으로 **검증 가능** | 프롬프트·재생성으로 보강 |
| **반응형** | 980px·좁은 뷰 **문서+목업** | 디바이스별 생성 권장 |
| **다크모드** | 목업 토글 + 토큰 명시 | 별도 생성·DS 권장 |
| **테이블 UX** | PRD Q4 갭 **명시적** | 프롬프트로 15건·필터 등 요청 가능 |
| **비용** | 문서·HTML 유지 | 키·시간·반복 호출 |
| **리스크** | 시각 정돈은 구현 단계 부담 | URL·도구 버전 **재현성** 관리 |
