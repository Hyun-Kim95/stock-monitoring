# 플랫폼 운영자 사이트 PRD (초안)

## 0. 정책 정합·누락 보완 (메타)

아래는 **본 PRD 초안**과 **기존 제품(설정 UI PRD·인증 구현·스키마·레포 규칙)** 을 대조해 드러난 **충돌·갭**을 한 곳에 모은 것이다. 세부 요구는 §2·§4~§11에 반영한다(§12는 변경 이력 전용).

| 구분 | 내용 |
|------|------|
| **세션의 “현재 테넌트” 단일성** | API `resolveAuthFromRequest`는 사용자의 **`memberships` 중 1건만**(생성일 오름차순 첫 건)을 세션 컨텍스트로 올린다. 즉 세션에 실려 오는 `tenantId`·`role`은 **“활성 워크스페이스”** 일 뿐, 플랫폼 운영이 대상으로 하는 테넌트와 **항상 일치하지 않는다**. 플랫폼 운영자 API·UI는 **요청마다 명시한 `tenantId`(또는 전역 전용 엔드포인트)** 를 기준으로 하고, **세션의 `tenantId`만으로 테넌트 스코프 쓰기·민감 조회를 수행하지 않는다**(실수·권한 우회 방지). |
| **`adminPre`·`ADMIN_API_TOKEN`과의 분리** | 설정 UI PRD에 따르면 `adminPre`는 **`Bearer`(`ADMIN_API_TOKEN`) 또는 세션+`OWNER`/`ADMIN`** 의 OR이다. 플랫폼 운영자 라우트를 여기에 **끼워 넣으면** Bearer만으로 테넌트 관리자와 **동일 수준의 보호 해제**가 될 수 있어 정책이 충돌한다. **요구:** 플랫폼 전용 라우트는 **`adminPre`와 분리된 검증**을 쓰고, **`ADMIN_API_TOKEN`만으로 플랫폼 운영자 권한을 통과시키지 않는다**. 자동화·스크립트가 필요하면 **`PLATFORM_API_TOKEN` 등 별도 환경변수 + 별도 프리핸들러**로 계약을 정한다(미사용 시 생략). |
| **테넌트 관리자 ≠ 플랫폼 운영자** | 동일 `User`가 특정 테넌트에서 `OWNER`/`ADMIN`이어도 **전역 운영자 플래그가 없으면** 플랫폼 운영자 사이트에 들어가면 안 된다(역방향도 마찬가지: 전역 운영자가 모든 테넌트의 설정 UI 권한을 갖는 것은 **본 PRD가 자동 부여하지 않음** — 필요 시 별도 정책). |
| **`SystemSetting` 이중 진입점** | `SystemSetting`은 테넌트 스코프로 **설정 UI(`/settings`)와 플랫폼 콘솔** 양쪽에서 바뀔 수 있다. 감사 로그(§4.4)에는 가능하면 **출처**(예: `SETTINGS_UI` / `PLATFORM_CONSOLE`)를 남겨 **누가 어느 경로로** 바꿨는지 구분한다. |
| **`StockQuoteHistory`와 테넌트** | `StockQuoteHistory`는 스키마상 **`tenantId` FK가 없고** `stockCode` 중심이다. “테넌트별 수집 건강도”는 **해당 테넌트 `Stock.code` 집합과의 조인·부분 집계**로 정의하거나, 글로벌 지표와 **혼동하지 않도록** API 계약에서 필드를 고정한다(§4.3 D-04·§9). |
| **HTTP 상태 코드** | **미로그인**은 **401**, 로그인했으나 전역 운영자가 아님은 **403**으로 통일한다. 웹은 리다이렉트 UX를 별도로 둘 수 있다. |
| **다크모드·모드 저장** | `.cursor/rules/40-dark-mode.mdc`에 맞춰 라이트/다크 전환과 **재방문 시 유지**(로컬 저장소 등)는 **일반 웹과 동일 원칙**을 따른다 — 플랫폼만 예외 두지 않는다(§4.1). |
| **웹 전용 범위** | `.cursor/rules/20-web-vs-app.mdc`에 따라 본 콘솔은 **웹(데스크톱 효율·테이블 중심)** 을 전제로 한다. **비목표:** 네이티브 앱 전용 운영 화면. |
| **디자인 게이트 예외 여부** | `.cursor/rules/65-design-gate.mdc`는 UI 변경에 이중안·승인을 요구한다. **내부 운영자 단일·최소 UI**에 한해 예외(와이어 단일안·Stitch 생략)를 둘지는 **승인 시점에 명시**하지 않으면 Gate 2·구현 착수 판단이 흔들린다 — §11에 **예외 조건 또는 “예외 없음”** 을 적어 둔다. |
| **테이블 규칙 vs 크로스 테넌트 목록** | `.cursor/rules/30-table-pagination.mdc`는 **테이블형 목록**에 적용한다. **사용자 검색(S-01)** 처럼 “검색 후 상세” 위주 화면은 **목록이 테이블이면** 동일 규칙을 적용하고, **단건·자동완성만** 인 경우 §4.5에서 **예외 전제**(건수 상한·운영 승인)를 문서화하지 않으면 DoD 해석이 충돌할 수 있다. |

## 문서 용어 (SSOT)

- **플랫폼 운영자 사이트**: **모든 테넌트**를 가로질러 **사용자 지원**과 **데이터·수집 운영**을 수행하기 위한 **관리자 전용 웹 UI** 및 이에 대응하는 **백엔드 API** 묶음. 최종 사용자용 대시보드·`/settings`와 **URL·권한 모델을 분리**한다.
- **테넌트 멤버십 역할** (`Membership.role`: `OWNER` / `ADMIN` / `MEMBER`): **해당 테넌트 워크스페이스** 안에서의 권한. 본 PRD의 “플랫폼 운영자”와 **동일하지 않다**.
- **플랫폼 운영자 권한**: `User`에 부여되는 **전역** 자격(구현 식별자는 예: `isPlatformAdmin`, `platformRole` 등 — 계약 문서 [`platform-operator-site/api-contract.md`](./platform-operator-site/api-contract.md)에서 확정). 본문에서는 **“전역 운영자 플래그가 참인 사용자”** 로 서술한다.
- **운영 인력 규모**: 정책상 **한 명**이나, 스키마·감사는 **복수 확장 가능**한 형태를 허용한다.

**관련 문서:** 테넌트 스코프 설정 UI는 [`settings-ui-prd.md`](./settings-ui-prd.md) (로그인 세션 + 해당 테넌트에서 `OWNER`/`ADMIN`). 본 PRD는 **크로스 테넌트** 운영 콘솔이다.

**문서 세트 (초기):** [`platform-operator-site/README.md`](./platform-operator-site/README.md) — `api-contract.md` · `operations.md` · `qa-checklist.md`. **SSOT는 PRD**이며, 정책 변경 시 PRD를 먼저 고친 뒤 세트 문서를 맞춘다(README 동기화 절차 참고).

## 1. 제품 목표

| 항목 | 내용 |
|------|------|
| **목표** | 다수 테넌트 환경에서 **가입 사용자 지원**과 **시세·뉴스 등 데이터·수집 관련 운영**을 한 곳에서 안전하게 수행한다. |
| **성공 지표** | 지원 건 처리 시간 단축, 테넌트 간 실수(잘못된 테넌트 수정) 감소, 민감 조회·변경의 **추적 가능성**(감사) 확보. |
| **비목표 (1차)** | 결제·청구 전체 콘솔, 최종 사용자 대시보드 기능 복제, 테넌트 `OWNER`가 쓰는 `/settings` 화면과의 UI 통합(별도 사이트·별도 권한 유지). |

## 2. 사용자·역할

| 역할 | 설명 |
|------|------|
| **플랫폼 운영자** | 전역 운영자 플래그가 있는 **가입 `User`**. 동일 DB·동일 로그인(OAuth/세션)을 사용한다. |
| **테넌트 관리자 (`OWNER`/`ADMIN`)** | 기존 제품 규칙대로 **자기 테넌트** 설정·종목 등을 다룬다. 플랫폼 운영자 사이트 접근 권한이 **자동으로 따라오지 않는다**. |

**인증 정책 (요구):**

- 플랫폼 운영자 사이트는 **로그인 세션(쿠키)이 유효하고**, 세션 주체 `User`에 **전역 운영자 자격**이 있을 때만 진입·API 호출이 허용된다. 브라우저 호출은 설정 UI와 같이 **`credentials: "include"`** 를 전제로 한다(헤더 토큰 정렬만으로 권한이 생기지 않게 서술·구현을 맞춘다).
- **미로그인** API: **401**. **로그인했으나 전역 운영자 아님**: **403**; 웹은 제품 홈 등으로 **리다이렉트**할 수 있다.
- **`ADMIN_API_TOKEN` / `adminPre`**: 플랫폼 운영자 보호에 **사용하지 않는다**(§0 메타). 테넌트 설정 쓰기 자동화가 필요하면 별도 토큰·별도 프리핸들러로 정의한다.
- 운영 정책은 **전역 운영자 계정 1개**이나, DB에는 향후 2명 이상을 허용할 수 있다(운영 절차로 1명 제한).

## 3. 현재 상태 (As-Is)

| 구분 | 내용 |
|------|------|
| **데이터 모델** | `Tenant`, `User`, `Membership`, `Stock`, `Theme`, `NewsSourceRule`, `SystemSetting`, `SupportInquiry`, `SupportInquiryReply`, `StockQuoteHistory` 등 — **테넌트 스코프**가 대부분의 비즈니스 엔티티에 존재한다. |
| **문의** | `SupportInquiry`는 사용자 작성 흐름과 저장이 가능하나, **플랫폼 운영자 전용 목록·상태·답변 API** 및 크로스 테넌트 UI는 본 PRD 범위에서 **신규 정의**가 필요하다. (스키마에 **문의 상태 enum**은 1차 생략 — §9 확정.) |
| **기존 “관리” UI** | 웹 `(admin)` 및 `/settings/*`는 **세션 + 해당 테넌트 컨텍스트의 `OWNER`/`ADMIN`** 전제에 가깝다. **모든 테넌트 열람**은 본 PRD의 전역 권한과 **분리**한다. |
| **세션에 올라오는 멤버십** | 현재 구현은 사용자당 **하나의 멤버십만** 세션 `tenantId`/`role`에 반영된다(다중 가입 시 나머지는 세션만으로는 드러나지 않음). 플랫폼 **사용자 상세(S-03)** 는 DB 기준 **전체 멤버십**을 보여야 하며, “세션에 보이는 테넌트 = 유일 테넌트”로 **오해하지 않도록** UI에 안내 문구를 둔다. |

## 4. 목표 상태 (To-Be) — 기능 요구사항

### 4.1 정보 구조 (IA)

- **1차 네비게이션(작업 중심 평면 메뉴, 2026-05-14 개정):**
  1. **대시보드** — 운영 핵심 KPI(테넌트 수·신규 회원·미답 문의·활성 공지 등) 한눈에. 진입 시 첫 화면.
  2. **회원** — 전역 사용자 **기본 목록(최근 가입 desc)** 과 검색을 함께 제공. 상세는 OAuth·멤버십.
  3. **문의** — **크로스 테넌트 통합 목록**(테넌트 컬럼·필터 포함). 상세는 본문·답변 + 답변 작성.
  4. **공지** — 운영자가 작성하는 **`Announcement`** 의 목록·작성·수정·발행·취소. 스코프는 전역 또는 테넌트(§4.6).
  5. **테넌트** — 기존 테넌트 허브 진입. 깊이 탐색(시스템 설정·카탈로그·시세 건강도)이 필요한 경우만 사용. 메뉴 위치는 4번째 이후로 내린다.
  6. **감사 로그** — `PlatformAuditLog` 검색·필터(§4.7).
- 메뉴는 “작업”으로 평면화하되 **데이터 스코프(테넌트)는 유지**한다. 크로스 테넌트 표·상세는 `tenantId`(또는 테넌트 요약)를 항상 컬럼·헤더에 노출해 잘못된 테넌트 액션을 방지한다.
- 진입 경로: `/platform`(웹) → 대시보드. 기존 `/platform/tenants` 리다이렉트는 폐기한다.
- **라이트/다크 모드** 및 **모드 지속 저장**(재방문·새로고침 후 일관)은 `.cursor/rules/40-dark-mode.mdc` 및 제품과 **동일 원칙**을 따른다.
- **상태 UI**(기본·로딩·빈·오류·권한)는 제품 공통 원칙을 따른다.

| **S-04** | **문의(`SupportInquiry`)**: 테넌트 필터·기간·키워드로 목록; 상세에서 본문·작성자·테넌트 확인. **2026-05-14 보강:** 기본 목록은 **크로스 테넌트**(모든 테넌트 통합)이며, 표에 `tenant` 컬럼·테넌트 필터를 둔다. 기존 테넌트별 목록(`/platform/tenants/:tenantId/inquiries`)은 깊이 탐색용으로 유지. | |

| **S-07** (신설, 2026-05-14) | **회원 기본 목록**: 검색어 없이도 **최근 가입 desc**로 페이지네이션된 사용자 목록을 보여준다. 검색은 기존 `users/search`와 동일 규칙 유지. | 운영자가 “최근 가입한 사용자”를 빠르게 확인. |

### 4.3 데이터·수집 운영

| ID | 요구사항 | 비고 |
|----|----------|------|
| **D-01** | **테넌트 목록·검색**: 이름·`tenantId`, 생성일. | 지원·수집 화면의 진입점. |

## 5. 개인정보·위험 작업

- 사용자 이메일·문의 본문 등은 **목적에 필요한 최소 범위**만 표시하고, 화면 캡처 유출을 고려한 **마스킹 옵션**을 디자인 단계에서 검토한다.
- `SystemSetting`·역할 변경·세션 무효화 등 **고위험 작업**은 확인 모달 + 감사 로그를 **필수**로 한다.

## 6. API·권한 (초안 원칙)

- **별도 라우트 접두사** 권장: 예 `/platform/...` 또는 `/ops/...` — 기존 `adminPre`(테넌트 `OWNER`/`ADMIN` + 선택적 `ADMIN_API_TOKEN`)와 **경로·프리핸들러 모두 분리**한다.
- **핸들러 검사**: `requirePlatformOperator` (가칭) — **세션 필수 + 전역 운영자 플래그**. `adminPre`와 **OR 결합하지 않는다**.
- **테넌트 스코프 데이터**는 요청에 **`tenantId` 명시**(쿼리 또는 헤더) — 세션에 실린 **단일 멤버십의 `tenantId`** 에만 의존하지 않는다(§0 메타).
- **자동화 전용 토큰**을 도입할 경우에만 `Authorization: Bearer …` 를 허용하고, 그때도 **테넌트 쓰기**는 요청 바디/쿼리의 `tenantId`와 **토큰 화이트리스트**를 계약에 명시한다(설정 UI PRD의 Bearer·세션 혼선 재발 방지).
- 상세 필드·에러 코드·상태 코드는 **[`platform-operator-site/api-contract.md`](./platform-operator-site/api-contract.md)** 에서 Gate 2 전 확정한다.

## 7. 화면 상태·오류

### 7.1 상태 모델 (UI)

| 상태 | 요구 |
|------|------|
| 기본 | 선택된 테넌트·필터가 명확히 보인다. |
| 로딩 | 스켈레톤 또는 스피너, **중복 제출 방지**(연타·더블클릭). |
| 빈 | 검색 결과 없음·목록 0건·테넌트에 종목 0건 등 **맥락별 안내 문구**를 구분한다. |
| 오류 | API 메시지·재시도; **401**은 미로그인, **403**은 권한 없음(전역 운영자 아님 포함)으로 일관. |
| 권한 | 전역 운영자 아님 → 진입 차단(웹 리다이렉트, API **403**). |
| **세션 만료** | 저장 중·폼 작성 중 세션 끊김 시 **401** 수신 → 로그인 유도, **입력 초안 손실** 가능성을 짧게 안내(복구 불가면 명시). |
| **네트워크/서버 불가** | 타임아웃·**503** 등은 재시도·서비스 점검 문구; 클라이언트는 **지수 백오프 또는 수동 재시도** 중 하나를 택해 계약에 명시한다. |

### 7.2 API 오류 응답 (요약 계약)

- **응답 형식**은 기존 공개 API와 동일한 **에러 봉투**(예: `{ error: { code, message } }`)를 따른다. `code`는 **기계 판별 가능한 짧은 식별자**, `message`는 운영자에게 읽히는 한국어(또는 제품 기본 언어)를 원칙으로 한다.
- **401** `UNAUTHORIZED`: 쿠키 없음·세션 만료·무효 토큰.
- **403** `FORBIDDEN`: 로그인됐으나 전역 운영자 아님(또는 자동화 토큰 권한 밖).
- **400** `VALIDATION_ERROR` 등: 필수 파라미터 누락(예: **`tenantId` 없음**), 잘못된 UUID 형식, 허용되지 않은 필터 조합, 본문 길이·형식 위반.
- **404** `NOT_FOUND`: 존재하지 않는 `tenantId` / `userId` / `inquiryId` 등 — **§9 확정:** UUID 단건은 **404**, 검색 목록은 **200 빈 배열**.
- **409** `CONFLICT`: 동시 편집·비즈니스 규칙 충돌(예: `SystemSetting` 갱신 시 **낙관적 잠금 실패** — §8.4).
- **429** `RATE_LIMIT`: 검색·목록·쓰기 남용 방지(운영 콘솔도 공격면이 됨). **`error.code` 명칭은 기존 API(`apps/api` 쓰기 레이트 리밋 등)와 통일**한다.
- **500/503**: 스택 트레이스·내부 DB 메시지를 클라이언트에 노출하지 않는다. **correlation id**를 응답 헤더 또는 본문에 두면 로그 대조가 쉽다(권장).

## 8. 엣지 케이스·오류 처리

### 8.1 테넌트·식별자

| 상황 | 기대 동작 |
|------|-----------|
| **`tenantId` 누락·빈 문자열** | **400** — 테넌트 스코프 엔드포인트는 명시 필수(§0·§6). |
| **`tenantId` 형식 오류** | **400**. |
| **존재하지 않는 `tenantId`** | **404** (§9 확정). |

| **테넌트에 활성 종목 0건** | D-04 건강도·요약은 **빈 상태**로 표시, 오류로 처리하지 않는다. |

| **답변 POST** | UI 연타 방지 + **60초 내 동일 본문 409 `CONFLICT`**(선택적 중복 방지) |
| **감사** | 비즈니스 쓰기와 **동일 트랜잭션** 내 `PlatformAuditLog` 기록(§8.7 a) |
| **세션 플래그 노출** | **`GET /auth/me`** 응답에 **`isPlatformOperator`** (`isPlatformAdmin` 동치) |
| **검색 상한** | **최대 100건** 클램프 + 응답 **`truncated: true`** 가능 |
| **페이지 범위** | `page` &lt; 1 → **1로 처리**; `page` &gt; `totalPages` → **200 + 빈 목록**, 응답 `page`는 **`totalPages`로 클램프** |
| **공지 정렬(목록)** (2026-05-14) | 기본 **`startsAt desc nulls last, createdAt desc`**. 상태 필터(`status`)·스코프 필터(`scope`)·기간 필터(`activeOn`)를 둔다. |
| **공지 게시 기간 검증** (2026-05-14) | `endsAt < startsAt`이면 **400 `VALIDATION_ERROR`**. 둘 다 없으면 상시 노출(다음 사이클 노출 UI 정책). |
| **감사 로그 보존** (2026-05-14) | 기본 **180일**(운영 합의 변경 가능). 자동 삭제 잡은 본 사이클 비포함. |
| **감사 기록 원자성** (2026-05-14) | §4.4 A-04 대상 액션은 비즈니스 쓰기와 **동일 Prisma 트랜잭션**에 `PlatformAuditLog` INSERT(§8.7 a 유지). 트랜잭션 실패 시 양쪽 모두 롤백. |
| **대시보드 KPI** (2026-05-14) | `GET /platform/overview` 응답: `tenantCount`, `newUserCountLast7d`, `inquiryUnansweredCount`, `announcementActiveCount`. 최근 실패 시세 등 추가 KPI는 가용 시 행 추가. |

## 10. 수용 기준 (MVP 요약)

1. 전역 운영자가 아닌 사용자는 플랫폼 운영자 사이트·API에 **접근할 수 없다**.
2. **사용자 검색 → 상세 → 멤버십 목록**이 동작한다.
3. **문의 목록(테넌트·기간 필터)** 및 **답변 등록**이 동작한다.
4. **테넌트 검색·선택** 후 해당 테넌트의 **`SystemSetting` 조회·갱신**이 동작한다.
5. 목록 화면은 **페이지당 15건·필터·페이지네이션** 정책을 만족한다(§4.5).
6. 플랫폼 운영자의 지정 작업에 **감사 로그**가 남는다(§4.4 A-01).
7. §8에 정의된 **대표 엣지**(미로그인·권한 없음·`tenantId` 누락·404 리소스·409 동시성·429)에 대해 API·UI가 **문서화된 코드와 화면**으로 대응한다.

## 11. 디자인·승인

- UI가 포함되므로 **디자인 산출물**(화면 구조, 기본/로딩/빈/오류/권한, 반응형, 다크모드) 승인 후 구현 착수한다(프로젝트 규칙 `.cursor/rules/65-design-gate.mdc`, `70-client-lifecycle-default.mdc`).
- **디자인 승인 = 구현 착수 승인**으로 본다.
- 이중 디자인안이 필요한 범위면: 안 A(로컬)·안 B(Stitch 또는 동등) 비교표 및 선택 기록을 본 문서 또는 `docs/design/`에 남긴다.
- **예외(명시 필수):** 내부 운영자 단일·최소 UI에 대해 **이중안 생략·와이어 단일안만** 허용할지 여부를 본 절 또는 승인 코멘트에 **“예외 없음” 또는 예외 조건**으로 남긴다. 미기재 시 `.cursor/rules/65-design-gate.mdc`의 기본(이중안·비교·선택 기록)을 따른다.
- **2026-05-12 확정:** **안 A** = 제품 사용자·`(admin)` 설정 셸과 동일 토큰·레이아웃(`admin-shell` 등). **안 B(Stitch)** 는 범위·일정상 생략 — 비교·사유는 [`docs/design/platform-operator-ui.md`](../design/platform-operator-ui.md) §11 표에 기록.
- **2026-05-14 추가:** §4.1 IA 재구성(작업 중심 평면 메뉴: 대시보드/회원/문의/공지/테넌트/감사 로그) 및 §4.6 공지·§4.7 감사 로그 신설은 **운영자 단일·최소 UI**로 분류해 **이중 디자인안(안 A·안 B) 비교를 생략**한다. 적용 근거: 운영자 콘솔의 비교 가치 대비 일정·인력 비용. 기존 `(admin)` 셸·토큰·레이아웃을 그대로 재사용하므로 시각 일관성 위험은 낮다. 후속 사이클에서 사용자 노출 UI(예: 공지 배너) 작업 시에는 본 예외가 자동 적용되지 않으며 별도 디자인 게이트를 거친다.
